# 功能-数据-代码映射分析报告

## 一、功能模块清单

1. **用户认证**：登录、注册、密码重置、邮箱验证
2. **战队管理**：创建战队、编辑战队、解散战队、战队资料查看
3. **成员管理**：邀请队员、审批入队申请、踢出队员、角色设置
4. **智能分组**：自动分组、分组批次管理、分组结果确认
5. **数据中心**：比赛记录管理、数据统计分析、训练计划管理、战队信息管理
6. **社区系统**：贴吧发帖、帖子浏览、评论、点赞、好友管理
7. **交易系统**：商品发布、商品浏览、商品管理
8. **赛事管理**：赛事创建、赛事浏览、赛事注册
9. **AI 助手**：智能聊天、战队管理建议

## 二、每个功能的详细映射

### 1. 用户认证

**前端页面**：
- `/app/auth/login/page.tsx`：登录页面
- `/app/auth/register/page.tsx`：注册页面
- `/app/auth/forgot-password/page.tsx`：忘记密码页面
- `/app/auth/reset-password/page.tsx`：重置密码页面
- `/app/auth/callback/page.tsx`：认证回调页面

**后端 API**：
- `/app/api/auth/check-email/route.ts`（POST）：检查邮箱是否已注册

**数据库表**：
- `profiles`：存储用户基本信息
- `auth.users`：Supabase 系统表，存储用户认证信息

**关键字段**：
- `profiles.id`：用户 ID（关联 auth.users.id）
- `profiles.email`：用户邮箱
- `profiles.nickname`：用户昵称
- `profiles.avatar`：用户头像

**数据流向**：
1. 用户在登录/注册页面输入信息
2. 前端发送请求到 Supabase Auth API
3. Supabase Auth 创建用户并触发触发器创建 profiles 记录
4. 前端获取用户信息并更新状态

**权限角色**：
- 所有用户（未登录用户也可访问认证相关页面）

### 2. 战队管理

**前端页面**：
- `/app/teams/new/page.tsx`：创建战队页面
- `/app/teams/edit/page.tsx`：编辑战队页面
- `/app/teams/[id]/profile/page.tsx`：战队资料页面
- `/app/teams/space/page.tsx`：战队空间页面

**后端 API**：
- `/app/api/team/dismiss/route.ts`（POST）：解散战队

**数据库表**：
- `teams`：存储战队基本信息
- `team_members`：存储战队成员信息

**关键字段**：
- `teams.id`：战队 ID
- `teams.name`：战队名称
- `teams.region`：服务器区域
- `teams.avatar_url`：战队头像
- `team_members.user_id`：成员用户 ID
- `team_members.team_id`：成员所属战队 ID
- `team_members.role`：成员角色

**数据流向**：
1. 用户在前端页面提交战队创建/编辑表单
2. 前端调用 Supabase API 插入/更新 teams 表
3. 对于创建战队，同时创建 team_members 记录将当前用户设为队长
4. 前端更新页面显示

**权限角色**：
- 队长：创建、编辑、解散战队
- 所有角色：查看战队资料

### 3. 成员管理

**前端页面**：
- `/app/teams/manage/page.tsx`：战队管理页面（成员管理）
- `/app/teams/applications/page.tsx`：战队申请页面
- `/app/teams/join/page.tsx`：加入战队页面

**后端 API**：
- `/app/api/applications/route.ts`（POST）：处理战队申请的审批和拒绝
- `/app/api/team/kick-member/route.ts`（POST）：踢出战队成员

**数据库表**：
- `team_members`：存储战队成员信息
- `team_applications`：存储入队申请信息
- `player_profiles`：存储队员游戏资料

**关键字段**：
- `team_members.user_id`：成员用户 ID
- `team_members.team_id`：成员所属战队 ID
- `team_members.role`：成员角色
- `team_applications.id`：申请 ID
- `team_applications.user_id`：申请人 ID
- `team_applications.team_id`：申请战队 ID
- `team_applications.status`：申请状态
- `player_profiles.user_id`：玩家用户 ID
- `player_profiles.team_id`：玩家所属战队 ID
- `player_profiles.game_id`：游戏 ID

**数据流向**：
1. 用户提交入队申请 → 前端调用 API 创建 team_applications 记录
2. 队长审批申请 → 前端调用 API 更新 team_applications 状态为 approved
3. API 同时创建 team_members 记录和 player_profiles 记录
4. 队长踢出队员 → 前端调用 API 删除 team_members 记录

**权限角色**：
- 队长、副队长：审批申请、踢出队员、设置角色
- 所有角色：提交入队申请

### 4. 智能分组

**前端页面**：
- `/app/group/page.tsx`：智能分组页面

**后端 API**：
- `/app/api/group/auto/route.ts`（POST）：智能分组功能
- `/app/api/group/batches/route.ts`（POST/GET/DELETE）：管理分组批次
- `/app/api/group/confirm/route.ts`（POST）：确认分组结果
- `/app/api/group/result/route.ts`（GET）：获取分组结果
- `/app/api/group/unlock/route.ts`（POST）：解锁分组批次
- `/app/api/group/route.ts`（GET）：获取分组相关信息

**数据库表**：
- `player_profiles`：获取队员游戏资料和位置信息
- `team_members`：获取战队成员列表
- `group_batches`：存储分组批次信息
- `group_members`：存储分组成员信息

**关键字段**：
- `player_profiles.user_id`：玩家用户 ID
- `player_profiles.main_positions`：主要位置
- `player_profiles.available_time`：可比赛时间
- `player_profiles.position_stats`：位置数据
- `group_batches.id`：批次 ID
- `group_batches.team_id`：战队 ID
- `group_members.batch_id`：批次 ID
- `group_members.user_id`：成员用户 ID

**数据流向**：
1. 队长进入智能分组页面 → 前端获取战队成员数据
2. 点击开始分组 → 前端调用 API 发送分组请求
3. API 分析队员数据并生成分组结果
4. 前端显示分组结果，队长可调整
5. 确认分组 → 前端调用 API 保存分组结果到数据库

**权限角色**：
- 队长、副队长：使用智能分组功能

## 四、缺失或冗余的映射

### 1. 缺失的映射

**智能分组功能**：
- 问题：智能分组 API 中使用了 `player_profiles` 表的 `main_positions` 字段，但在实际代码中，`main_positions` 可能存储为 JSONB 或 TEXT[] 类型，需要确保前端传递的数据格式正确

**交易系统**：
- 问题：`team_sales` 表缺少 `seller_id` 字段，无法关联卖家信息，导致前端显示 "未知用户"

**成员管理**：
- 问题：`team_members` 表缺少 `join_method` 字段，但在 API 代码中尝试插入该字段

### 2. 冗余的映射

**数据库表**：
- `team_applications` 表中的游戏资料字段（如 `game_id`、`current_rank` 等）在审批后会复制到 `player_profiles` 表，存在数据冗余

**API 路由**：
- `/app/api/group/route.ts` 和 `/app/api/group/batches/route.ts` 功能有重叠，可能存在冗余

## 五、数据流图（文字描述）

### 1. 用户注册流程

1. **用户操作**：用户在注册页面输入邮箱、密码和其他信息
2. **前端处理**：`/app/auth/register/page.tsx` 验证表单数据
3. **API 调用**：前端调用 Supabase Auth 的 `signUp` 方法
4. **数据库操作**：
   - Supabase Auth 创建 `auth.users` 记录
   - 触发器自动创建 `profiles` 记录
5. **返回前端**：Supabase 返回用户信息
6. **前端更新**：前端更新用户状态，跳转到首页

### 2. 队长审批入队申请流程

1. **用户操作**：队长在战队申请页面点击 "批准" 按钮
2. **前端处理**：`/app/teams/applications/page.tsx` 收集申请信息
3. **API 调用**：前端调用 `/app/api/applications/route.ts`（POST），action 为 "approve"
4. **数据库操作**：
   - 更新 `team_applications` 表中对应申请的状态为 "approved"
   - 在 `team_members` 表中创建新记录，角色为 "队员"
   - 在 `player_profiles` 表中创建或更新队员游戏资料
5. **返回前端**：API 返回成功信息
6. **前端更新**：前端更新申请列表，显示批准状态

### 3. 智能分组流程

1. **用户操作**：队长在智能分组页面点击 "开始分组" 按钮
2. **前端处理**：`/app/group/page.tsx` 收集战队 ID 和锁定成员信息
3. **API 调用**：前端调用 `/app/api/group/auto/route.ts`（POST）
4. **数据库操作**：
   - 从 `team_members` 表获取战队成员列表
   - 从 `player_profiles` 表获取队员游戏资料和位置信息
   - 分析数据并生成分组结果
5. **返回前端**：API 返回分组结果
6. **前端更新**：前端显示分组结果，队长可调整
7. **用户操作**：队长点击 "确认分组" 按钮
8. **前端处理**：收集最终分组结果
9. **API 调用**：前端调用 `/app/api/group/confirm/route.ts`（POST）
10. **数据库操作**：在 `group_batches` 和 `group_members` 表中保存分组结果
11. **返回前端**：API 返回成功信息
12. **前端更新**：前端显示分组完成信息

### 4. 发布招募流程

1. **用户操作**：队长在发布招募页面填写招募信息
2. **前端处理**：`/app/teams/recruit/page.tsx` 验证表单数据
3. **API 调用**：前端直接调用 Supabase API
4. **数据库操作**：在 `team_recruits` 表中创建新记录
5. **返回前端**：Supabase 返回成功信息
6. **前端更新**：前端显示发布成功信息，跳转到招募管理页面
